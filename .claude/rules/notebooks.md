# 가이드 노트북 작성 스타일

01~03을 만들며 사용자 피드백으로 수렴한 스타일. 새 노트북도 이 톤을 유지한다.

## 구조

- 섹션 제목은 내용 그대로: "1. Ingest", "2. 검색". 비유·연출(N막, 수업, 여정) 금지.
- 흐름: 짧은 마크다운 도입 → 코드 셀(몇 줄) → 실제 출력 → "출력에서 볼 것" 해석 셀.
- 시나리오는 실제 데이터(LoCoMo 대화)로 구성하고, 설계 단계에서 dry-run으로
  예상 동작을 실측 검증한 뒤 본문을 쓴다. 머리로 예측한 시나리오는 틀린 적 있음.

## 설명

- 셀당 몇 줄. 긴 설명은 신호가 아니라 소음.
- 손계산 금지 — 숫자는 코드가 계산해서 출력으로 보여준다.
- 출력이 말하게: 해석 셀은 "출력의 어디를 보라"고 짚는 역할까지만.
- 설명 추가는 사용자 질문이 드러낸 구멍에만. 미리 다 설명하려 들지 않는다.
  (실례: "conv-26이 뭔데?", "세그먼트가 맞아? 페이지 아니야?" → setup 셀과
  용어 첫 등장에 한 줄 설명을 추가하게 됐다.)
- 분량 불리기 금지: 함정 절, 요약 퀴즈, 관습적 부록 넣지 않는다.

## 언어

- 기술 용어는 영어 그대로 (eviction, consolidation, retrieval, heat,
  promotion...). 번역 조어(되뇌기, 응고)·비유 금지.
- 코드 식별자는 영어. 출력 라벨·주석·마크다운 본문은 한국어 OK.

## 기계적 규칙 (사고 이력 있음)

- 셀 source/outputs는 **줄 단위 리스트** 유지. 문자열을 리스트처럼 순회하면
  문자 단위로 쪼개진다. 복구: source를 join 후 `splitlines(keepends=True)`.
- 출력 달린 코드 셀을 고치면 반드시 재실행:
  `uv run jupyter nbconvert --to notebook --execute --inplace notebooks/NN_*.ipynb`
- 재실행에 LLM 호출이 들어가면 호출 수·예상 시간을 먼저 알리고 합의 후 실행.
